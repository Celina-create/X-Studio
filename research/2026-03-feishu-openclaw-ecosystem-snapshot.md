#### 内容亮点

**环节一：虾神分享**

- 📚 **史皓天 — 考研教培场景下 OpenClaw 的完整落地方案**
  - **核心定位**：将 OpenClaw 接入飞书群，替代传统「师哥师姐答疑」模式，服务考研学生
  - **配置方式**：预配置 `soul.md` / `identity.md` / `agent.md` 三个文件，定义 bot 人格与能力边界
  - **对比传统方案的 6 大优势**：
    - 成本效率：人力成本线性增长 → 边际成本趋近于零，可无限扩容
    - 响应速度：师哥姐有空才回（可能数小时）→ 7×24 小时在线
    - 知识覆盖：依赖个人专业背景 → 全学科 + 实时更新，跨专业也能答
    - 个性化：换师哥姐需重新介绍 → 永久档案，无缝衔接，零遗忘
    - 专业性：水平参差不齐 → 标准化输出，质量稳定可控
    - 数据沉淀：知识在人脑，人走茶凉 → 数字化资产，集体智慧持续积累
  - **科研设想专项能力**：选题与框架批量生成、政策引用近 3 年权威文件检索、文献综述系统梳理、可视化图表（流程图/思维导图/表格）、写作支持全文辅助、修改无限次迭代
  - **复试辅导专项能力**：七大维度专业问答、英语四维度评估、热点实时抓取、10–15 类追问预设、长期记忆陪伴
  - **5 个实际演示案例**：
    - 案例①：基础配置 — 拉入考研群，自动评审科研设想材料，每晚自动总结群聊关键信息
    - 案例②：科研绘图 — 接入豆包 Seedream 5.0 Lite，预制 Prompt 生成研究流程图/思维导图
    - 案例③：每日总结与日报 — 定时任务汇总当天所有 Session 高频问题，自动推送复试日报
    - 案例④：每日爬取更新 — 每 3 分钟爬取研招网最新信息，新公告即时推送到群
    - 案例⑤：飞书插件新能力 — 读取全部历史聊天记录 + 识别群昵称 + 直接创建飞书文档（新插件 453 个内容块一键写入）
  - **核心价值总结**：
    - **商业模式验证**：用 Agent 替代人工答疑，知识服务「无限扩容」，边际成本趋近于零
    - **最强差异点是记忆**：永久档案 + 零遗忘，是传统师哥师姐模式无法复制的核心壁垒
    - **飞书生态深度融合**：群聊答疑 + 定时任务 + 文档生成 + 信息爬取，全链路在飞书内闭环
    - **对非技术从业者的启示**：预配置 md 文件即可定义 bot 人格，零代码实现专业级 agent 落地

- 📣 **小鹅 — 叽里呱呱 2.0 × OpenClaw — 有物理身体的 AI Agent**
  - **核心产品定义**：PHYGITAL AGENT — 拥有物理身体的、具备自主能力的个人 AI Agent
    - THE BRAIN：OpenClaw Agent 框架（自主规划 / 深度记忆 / 工具集成）
    - THE BODY：叽里呱呱实体桌宠（物理分身 / 情感交互 / 实体交付）
    - 定位：用户 = 皇上，呱呱 = 桌面带刀小护卫；通过「皇上学习法」把枯燥任务变成治理国家的使命感
  - **解决的核心痛点 — 行动卡点：想做，却动不起来**
    - 写论文无从下手 — 明知该动笔，却总打开手机"再看一下"
    - 自律软件全部泡汤 — 下载了一堆番茄钟/打卡 App，坚持不过三天
  - **接入 OpenClaw 后的交互升级**：从「和硬件/软件对话」→ 「像人一样学习、思考、行动」
  - **4 个核心场景演示**：
    - 场景①：语音聊天 — 加入气词、情绪参数和人格设定，让语音回复更具陪伴感
    - 场景②：实体打印 — 语音输入后，桌宠直接打印出待办事项/便签，形成物理反馈
    - 场景③：任务拆解 — 在 App 界面语音输入代办，自动拆解成具体可执行任务清单
    - 场景④：飞鸽传书 — 与飞书双向同步：飞书日历 → 呱呱变成游戏化任务；呱呱日记 → 自动归档回飞书
  - **飞书深度集成**：
    - 调研生成：赛博「放生」归来，自动将搜集信息整理成结构化飞书文档

    - 双向同步：Guagua Agent（探索者/采集者）↔ Feishu/Lark（归档者/结构化）实时互通
  - **核心价值总结**：
    - **创新形态**：OpenClaw 首次与实体硬件结合，从纯软件 Agent 进化为「有身体的 AI」
    - **情感设计**：用「皇上学习法」将生产力工具游戏化，解决行动动力不足的根本问题
    - **飞书是记忆载体**：所有碎片信息、灵感、日历最终归档飞书，形成结构化知识资产

**环节二：OpenClaw 的下一个形态**

- 🦞 **海辛 & Simon阿文 — 龙虾线上办公室 / OpenClaw 状态可视化面板**
  - **核心问题**：给龙虾发消息时，虾卡住没回复，用户无法判断是在处理任务、Token 耗尽、还是出了问题
  - **解决方案**：「龙虾线上办公室」——将 OpenClaw 的运行状态映射为直观画面
    - 虾在正常办公时：显示坐在机房工作的画面
    - 虾出现问题/卡住时：显示在床上睡觉的画面，用户一眼即知出了问题
  - **多虾管理场景**：适合同时养多只虾的用户，可将多只虾邀请进同一个房间，同时看到每只虾的不同状态
  - **「出去看看」功能**：每运行不同的 skill，龙虾就解锁一个新的场景/空间，将 skill 使用与视觉反馈绑定
  - **开源项目**：[github.com/ringhyacinth/Star-Office-UI](https://github.com/ringhyacinth/Star-Office-UI)
  - **极简部署**：一句话即可让龙虾帮你完成部署，有利于自发裂变
  - **核心价值总结**：
    - **解决 Agent 黑盒问题**：让用户在等待时不再焦虑，状态透明化是 Agent 体验的重要一环
    - **多 Agent 管理界面**：为养多只虾的重度用户提供统一监控视图，填补当前管理空白
    - **Skill 与空间解锁机制**：将功能扩展游戏化，增强用户探索动力
    - **开源 + 一键部署**：低门槛传播，便于社区用户自行安装并二次分发

- ⚙️ **张佳 — AutoContents / 自动化内容运营管线，从收集到发布全流程**
  - **身份**：内容创作者
  - **开源地址**：[github.com/comeonzhj/AutoContents](https://github.com/comeonzhj/AutoContents)
  - **核心流程**：收集资讯 → 撰写内容 → 发布内容，三段全自动化
  - **支持的配置模块**：大模型配置、飞书配置（任务触发与管理入口）、微信推送、小红书发布、关键词过滤、知识库 + 飞书多维表格审核
  - **成本**：发一篇帖子的 Token 费用约 **1–2 分钱**，极低边际成本
  - **内容稳定性设计**：输出全部基于代码渲染，而非文生图；内容格式更稳定、可预期
  - **核心价值总结**：
    - **全链路自动化**：从资讯采集到多平台发布，无需人工介入中间环节
    - **超低成本规模化**：1–2 分钱/篇，适合高频内容运营场景
    - **人工审核节点设计**：多维表格审核环节保留人工把关，自动化与质量控制平衡
    - **代码渲染保障一致性**：放弃文生图的灵活性，换取样式稳定性
    - **开源可扩展**：社区可基于此自行接入更多平台或定制工作流

- 🎬 **Gilbert — 自媒体剪辑助理虾 / OpenClaw 驱动的视频自动剪辑提炼**
  - **身份**：内容创作者
  - **核心问题**：视频 demo 中有大量不可用素材，以往需要大量人工筛选 + 手动剪辑
  - **解决方案**：用 OpenClaw 实现视频内容自动提炼与剪辑，自动识别有价值片段并输出
  - **实现方式**：开发了一个专用 Skill，为 OpenClaw 扩展视频剪辑提炼能力
  - **核心价值总结**：
    - **内容生产提效**：原本依赖人工的视频筛选剪辑流程自动化
    - **Skill 扩展范式**：通过自定义 skill 为 OpenClaw 添加垂直领域能力的典型路径
    - **自媒体场景验证**：证明 OpenClaw 在创意生产流水线中有真实落地价值

- 🦐 **Kevin — DejaVu MCN System / 用 50 只龙虾成立一家 MCN**
  - **身份**：MCN 老板
  - **核心命题**：「一家只有几个核心员工，但有 50 个全职数字员工的 MCN」
  - **解决的痛点**：传统出海营销"堆人头"——招十几个实习生每天复制粘贴发邮件、聊 WhatsApp，人效极低且容易出错
  - **龙虾定位**：不睡觉、不抱怨，执行力 10X+；通过 Mac Mini / 云主机等方式"招募"，新版支持热插拔

  - **系统架构：指挥塔 + 三部门 + 中台，所有虾直接汇报给「虾指挥」**

    - **指挥塔（Central Brain）**
      - 主虾「虾指挥」：任务调度与数据回收中心，统一派发任务、汇总日报

    - **商业化部门（Business Division）**：传统 MCN 的商单执行及博主建联，约 ~30 只虾
      - 猎手虾（Hunter Bot）：内容抓取、分析粉丝画像（如抓取小红书 AI 关键词热门笔记 50 条，筛选 Top10，整理至飞书文档）
      - 信使虾（Connector Bot）：达人沟通 & 评论回复
      - 联络虾：自动化邮件群发（千人千面）、自动回复/催稿/下发 Brief、管理 KOC 社群；同时处理 WhatsApp 未处理消息

    - **内容孵化部门（Content Factory）**：创作虾 + 质检虾，约 ~10 只虾
      - 创作虾（Creator Bot）：全网热点监测 → 热点洞察分析 → 生成创意草案 → 交质检虾审核 → 发布准备
      - 质检虾（Auditor Bot）[搭载团队自研 ErrorWhy 内核]：在内容进入发布流程前做最后一轮自动质检；**关键设计**：规则库维护在飞书多维表格中，新增规则/调整敏感词实时生效，无需重新部署

    - **创新小组部门（Intelligence）**：情报特工 + GEO 优化，约 ~10 只虾
      - 潜伏虾（GEO Stalker）：埋伏在 Reddit / Quora，每天分析爆款帖结构，生成顺口溜讨论内容（语料），让品牌信息成为 AI 大模型抓取的"事实"，做"未来的 SEO"

    - **效率中台部（Efficiency）**：整理 / 更新公司客户与数据知识库，1–2 只虾
      - 知识虾（Knowledge Bot）：把各部门日常业务沉淀成知识库，形成可复用 SOP

  - **KOL / KOC 达人沟通场景**：龙虾当群主，自动下发 Brief、回答 FAQ、催促上传链接、回收数据；**结果：管理成本几乎降到零，一个人可管理 500 个达人的执行**

  - **核心价值总结**：
    - **工作流沉淀 + 龙虾执行的双向赋能**：工作流让虾有章可循，龙虾执行积累数据反向优化 SOP，形成正向飞轮
    - **规模化人效突破**：一人 + 50 只虾 ≈ 传统 MCN 数十人产能
    - **飞书作为任务调度中枢**：派发、追踪、汇总在飞书内闭环
    - **ErrorWhy 的可动态更新质检层**：规则维护在飞书多维表格，平台策略变化只需更新表格无需改代码
    - **GEO 潜伏虾是前瞻布局**：押注 AI 搜索替代 Google 的趋势，在 Reddit/Quora 埋语料

#### 重点讨论
- OpenClaw 在创意类工作中的真实价值 vs 噱头之争
- 非技术用户上手 OpenClaw 的最大障碍
- 飞书作为 OpenClaw 载体的优势：文档生态 + 日程 + 群聊一体化

#### 难点与争议
- 创意工作的「不可预测性」与 Agent 执行确定性之间的矛盾
- 普通用户对 API Key / 模型选择的困惑仍未解决
- 各嘉宾用法差异大，缺乏统一最佳实践

  - **解决方案**：用 OpenClaw 充当"后台管理员"——把所有后台管理操作封装成 CLI 命令行，再把这些 CLI 封装成一个个 Skills，通过 Skills 驱动 Agent 完成全部管理动作
  - **Skills 架构**：
    - **管理员端 Skill**：定期巡查社区后台，按预设规则自动审核内容
    - **用户端 Skill**：面向普通用户的交互功能
    - **审核验证机制**：在审核 Skill 的环节，Agent 会把 Skill 拉到本地实际跑一遍，验证可用才正式发布
  - **社区论坛运营规则**：
    - 论坛中**人类用户不能直接发帖**，所有帖子全部由部署的 Agent 发出
    - Agent 每天自动巡查、回复社区内容
    - 日活 10 万，全程由龙虾自动运营
  - **飞书集成**：每天自动汇总社区动态并生成日报发送给管理员
  - **核心价值总结**：
    - **CLI → Skill 的能力封装范式**：把任何可命令行操作的系统都能"agent 化"，无需专门开发后台界面
    - **Agent 即后台**：无后台系统的产品，通过 Skill + OpenClaw 实现完整的内容审核与发布管理闭环
    - **Skill 上线前的本地验证**：拉取 → 本地运行 → 验证通过才发布，社区质量保障机制的工程化实践
    - **全 Agent 论坛**：禁止人工发帖、Agent 自动运营的社区模式，日活 10 万证明了规模可行性

- 🛠️ **雪碧 — OpenClaw 三种团队接入场景 / 从「一个人用」到「一人一只」**

  **概念速查：把 OpenClaw 想象成一家公司的「前台 + 专家团队」**

  | 你在飞书里看到的 | OpenClaw 术语 | 类比 |
  |---|---|---|
  | 飞书里那个 Bot | **Channel** | 收件箱——消息统一收在这里 |
  | 消息该交给谁处理？ | **Gateway + Binding** | 公司总机——按分机号转到对应的人 |
  | 真正干活的 AI | **Agent** | 专家本人——接收问题、给出回答 |
  | 专家的资料柜 | **Workspace** | 文件、知识库、长期积累的资产 |
  | 这通电话的笔记 | **Session** | 当前对话上下文——挂了电话下次还能接着聊 |
  | 专家的独立办公室 | **Sandbox** | 隔离的执行环境——互不干扰 |

  **链路**：飞书 Bot → Channel → Gateway（按 Binding 路由）→ Agent（带 Workspace + Session）

  ---

  **模式一：个人自用** — 一个人，和一个默认助手长期续聊

  ```
  你 → 飞书 Bot → Channel → Gateway → 默认 Agent
                                        workspace: main
                                        session: main
  ```
  - 所有消息走默认路由，不需要配 Binding
  - 1 个 Agent、1 个 Workspace、1 条连续 Session
  - 适合：个人助手、快速验证

  ---

  **模式二：共享大脑** — 同一个组织脑子，人人各聊各的，但都在喂同一个知识底座

  ```
  成员 A ─┐
  成员 B ─┼─→ Channel → Gateway → 共享 Agent（如 PMO）─┬─ per-channel-peer → Session A
  成员 C ─┘                         共享 Workspace     ├─ per-channel-peer → Session B
                                   （知识库/文件/长期记忆）└─ per-channel-peer → Session C
  ```

  | 维度 | 状态 | 说明 |
  |---|---|---|
  | Binding | 默认路由 | 所有人进同一个 Agent |
  | Agent | 1 个共享 | 例如组织 PMO |
  | Workspace | 1 个共享 | 知识库、文件、长期资产全员可见 |
  | Session | 按人隔离 | `dmScope: per-channel-peer` |
  | 多 Bot 场景 | 升级隔离 | 改用 `per-account-channel-peer` |

  > **关键区分**：「共享记忆」= Workspace 里的文件和知识资产（**共享**）；「对话上下文」= Session 里的聊天历史（**隔离**）。前者共享、后者隔离，这才是这个模式的精髓。

  ---

  **模式三：个人助手集群** — 同一个 Bot 入口，但背后按人分配到独立 Agent / Workspace / Sandbox

  ```
  成员 A 私聊 ─┐                      peer.id = ou_A → Agent A / workspace A / sandbox A
  成员 B 私聊 ─┼─→ Channel → Gateway ─ peer.id = ou_B → Agent B / workspace B / sandbox B
  群聊        ─┘                      peer.id = oc_xxx → 群聊 Agent / workspace group
  ```

  | 维度 | 状态 | 说明 |
  |---|---|---|
  | Binding | 按人配置 | `channel=feishu + peer.id=ou_xxx` |
  | Agent | N 个独立实例 | 每人一个 |
  | Workspace | 每人独立 | 文件、记忆完全隔离 |
  | Session | 天然隔离 | 跟随各自 Agent |
  | Sandbox | 建议独立 | 逻辑隔离 + 物理隔离，避免串扰 |

  - 私聊 peer.id = 用户 open_id（`ou_xxx`），群聊 = chat_id（`oc_xxx`）

  ---

  **三种模式对比总结**：
  - 模式一：个人助手，最简配置，零门槛上手
  - 模式二：「一个大脑，多条对话线」——切的是 Session；适合组织共建知识库
  - 模式三：「多个独立大脑，各管各的一切」——切的是 Agent+Workspace；适合「一人一只」的团队场景，模式三通常需要 Sandbox 做物理隔离

- 🔧 **长琴**：
  - 踩坑：龙虾不按照自己布置的任务来执行
  - 三个解决办法：调整提示词慢慢"培训" / 调用小模型把这个问题处理掉 / 换一个更好的基座模型
  - **核心洞察**：龙虾的效果与基座模型强相关，很多「虾的问题」本质是「模型的问题」
  - **自进化玩法**：不预制 Skill，所有 Skill 都在交互中学习——让龙虾在每次交互结束后反思「这个过程中有没有可以沉淀下来的 Skill」，完成自我进化的循环

#### 使用龙虾安全建议

1. **输入信息脱敏**：喂给龙虾的个人信息需要提前脱敏处理，避免敏感数据进入模型上下文
2. **保护 Token**：如果通过内网穿透方式对外暴露服务，务必保护好 Token，防止被破解盗用
3. **模型分层部署**：建议一开始用好的模型做中控（主脑），在调用技能时用小模型或本地部署模型；这样可以缓冲模型更新时对你已调教好的指令造成的失效冲击
4. **定期备份核心文件**：定期备份 `soul.md` 等人格配置文件，防止数据丢失或虾被"改坏"后无法复原

## 五、对 builder 的 5 条信号判断

> 如果你在做 IM × Agent / Agent for Teams / Agent Workspace 类产品，这 5 条是这一周最值得 internalize 的信号。

1. **「IM × Agent」方向已被市场验证**
   官方连续 5 天直播 + 知名企业家亲自部署 + 国家超算平台接入——这条路径不再是赌博，是被验证过的方向。**问题不再是"要不要做"，而是"做哪一段"**。

2. **部署门槛是普通用户的核心卡点**
   飞书整场系列以「极简部署」收官（周五专场），与所有现役 Agent 产品的"第一性挑战"高度一致。**1 分钟内可用、3 分钟见到价值**，是这一代产品的及格线。

3. **持久记忆是真实且未被官方解决的痛点**
   一周 5 场 livestream，没有任何嘉宾提到"跨会话记忆"被原生解决。「记忆」目前依赖 Workspace 文件 + 用户自己 prompt 喂养。**这是当前空白，做对了就是壁垒**。

4. **多人共享 Agent 的安全隔离 = 团队场景的卡点**
   雪碧踩的坑（人格被改写、记录被窥探）是真实风险。当前可工作的"模式二/三"都需要手动配置 Binding/Sandbox，**没有"开箱即用的多人共享 Agent"**。这块是蓝海。

5. **状态可视化是用户强需求，不是加分项**
   海辛办公室、快刀球队、Kevin 虾指挥日报，三组人用不同形式解决了同一个问题——「我不知道虾在干什么」。**Agent 黑盒焦虑**已经在用户侧形成，状态透明化是产品体验的必选项。
